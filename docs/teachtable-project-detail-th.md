# TeachTable: รายละเอียดโปรเจกต์ฉบับมืออาชีพ

## 1. บทสรุปผู้บริหาร

TeachTable คือระบบบริหารตารางเรียนและตารางสอนสำหรับสถานศึกษา ที่ออกแบบมาให้ใช้งานจริงในบริบทโรงเรียน โดยรวม `master data`, `planning workspace`, `validation`, `collaboration` และ `document export` ไว้ในระบบเดียว ผู้ใช้สามารถเริ่มจากการจัดการข้อมูลครู ห้องเรียน รายวิชา และแผนรายวิชา จากนั้นจึงเข้าสู่หน้าจัดตารางเพื่อวางคาบแบบกึ่งอัตโนมัติ ตรวจสอบความชนกันของครู ห้อง และกลุ่มผู้เรียน ก่อนส่งออกเป็น `CSV` และ `PDF A4`

จุดเด่นของโปรเจกต์นี้คือไม่ได้เป็นเพียง demo UI แต่มี workflow ที่เชื่อมกันครบตั้งแต่การเข้าสู่ระบบด้วย Firebase, การโหลดข้อมูลกลาง, การจัดตารางอัตโนมัติ, การล็อก slot สำหรับการทำงานหลายคน, การเก็บ version ของตาราง และการพิมพ์หรือส่งออกเอกสารที่พร้อมใช้งานจริง

## 2. เป้าหมายของระบบ

- ทำให้ฝ่ายวิชาการเห็นภาพรวมความพร้อมของตารางสอนจากหน้าจอเดียว
- ลดความผิดพลาดจากการจัดครูชนกัน ห้องชนกัน หรือจัดคาบไม่ครบ
- รองรับรูปแบบการสอนที่ซับซ้อนกว่าระบบพื้นฐาน เช่น `split group` และ `team teaching`
- ช่วยให้หลายคนทำงานบนตารางเดียวกันได้โดยไม่เขียนทับกัน
- ส่งออกเอกสารในรูปแบบที่พร้อมนำไปใช้งานหรือพิมพ์แจกได้ทันที

## 3. แนวคิดการออกแบบและ Design Direction

### 3.1 ภาพลักษณ์ของงานออกแบบ

ดีไซน์ของ TeachTable อยู่ในกลุ่ม `operational product UI` มากกว่าหน้าเว็บเชิงการตลาด โดยให้ความสำคัญกับการอ่านง่าย การตัดสินใจเร็ว และการใช้งานต่อเนื่องในงานธุรการจริง

ลักษณะเด่นของ visual design ใน implementation ปัจจุบันมีดังนี้

- ใช้ฟอนต์ไทย `Prompt` สำหรับหัวเรื่อง และ `Sarabun` สำหรับเนื้อหา เพื่อให้ภาพลักษณ์ร่วมสมัยแต่ยังเหมาะกับงานเอกสารทางการ
- ใช้โทนสีหลัก `ฟ้า-เขียว` เป็นแกนของระบบ สื่อถึงความน่าเชื่อถือ ความเป็นระบบ และความรู้สึกสะอาดตา
- ใช้พื้นหลังแบบ `soft gradient + grid texture` ทำให้ระบบดูทันสมัยโดยไม่รบกวนข้อมูล
- ใช้ panel โปร่งแสงเล็กน้อยร่วมกับ `blur` และเงานุ่ม เพื่อสร้างความรู้สึกเป็น workspace ระดับ production มากกว่าหน้าฟอร์มพื้นฐาน
- ใช้มุมโค้งขนาดใหญ่และ spacing ที่โปร่ง ทำให้ระบบดูเป็นมิตร แม้ข้อมูลจะหนาแน่น

### 3.2 หลักคิดด้าน UX

ระบบถูกออกแบบภายใต้แนวคิด `อ่านง่าย ควบคุมง่าย ตัดสินใจเร็ว` ซึ่งเห็นได้ชัดจากองค์ประกอบต่อไปนี้

- หน้า dashboard สรุปตัวเลขสำคัญและสถานะความพร้อมทันที
- ข้อมูลถูกจัดเป็นกลุ่มหน้าที่ชัดเจน ไม่บังคับให้ผู้ใช้กระโดดหลายหน้าจอโดยไม่จำเป็น
- หน้าจัดตารางรวม board, กลุ่มคาบที่ยังไม่ลง, คำแนะนำ, ผู้ใช้งานออนไลน์, lock และ validation ไว้ใน workspace เดียว
- ปุ่มหลัก เช่น `จัดอัตโนมัติ`, `ตรวจสอบ`, `ส่งออก CSV`, `ส่งออก PDF`, `พิมพ์`, `บันทึก` อยู่ในตำแหน่งที่คาดเดาได้
- ใช้โทนสีและ pill labels เพื่อสื่อสถานะ เช่น พร้อมใช้งาน, warning, error, ภาระครูใกล้เต็มเพดาน

### 3.3 Motion และ Responsive Design

แม้แอปจะเน้นการใช้งานเชิงระบบ แต่ยังมี motion ที่ช่วยเพิ่มคุณภาพประสบการณ์ใช้งาน

- ใช้ animation `float-up` กับ boot screen และ auth area เพื่อให้การเปิดระบบดูมีจังหวะ
- ใช้ hover และ transition กับปุ่ม, chips และ panels เพื่อสร้าง feedback ที่ชัด
- รองรับ responsive layout โดยเมื่อหน้าจอแคบลง ระบบจะปรับ sidebar, dashboard grid, chart grid และ workspace grid ให้เรียงเป็นคอลัมน์เดียว
- Timetable workspace ถูกออกแบบให้ยุบจากหลาย panel เป็น stack บนหน้าจอขนาดเล็กได้

## 4. โครงสร้างเมนูหลักของระบบ

Navigation หลักของ TeachTable มี 4 เมนู

- `ภาพรวม`
- `ข้อมูลหลัก`
- `ตารางสอน`
- `ตั้งค่า`

โครงสร้างนี้สะท้อน workflow จริงของโรงเรียนอย่างชัดเจน คือเริ่มจากการดูสถานะรวม, เติมข้อมูลพื้นฐาน, จัดตาราง และปิดท้ายที่การตั้งค่าเอกสารหรือเอกลักษณ์สถานศึกษา

## 5. อธิบายหน้าจอหลักทีละส่วน

### 5.1 Boot Screen

เป็นหน้าจอเริ่มต้นที่ใช้ระหว่างระบบกำลังตรวจสอบสิทธิ์และโหลดข้อมูลล่าสุดของสถานศึกษา จุดประสงค์คือให้ผู้ใช้รับรู้ว่าระบบกำลังเตรียม workspace ไม่ใช่หน้าค้างหรือโหลดผิดพลาด

องค์ประกอบสำคัญ

- ชื่อผลิตภัณฑ์ `TeachTable`
- หัวข้อสถานะการเริ่มต้นระบบ
- ข้อความอธิบายว่ากำลังตรวจสอบสิทธิ์และโหลดข้อมูล

### 5.2 Authentication Screen

หน้าล็อกอินของระบบถูกออกแบบเป็น 2 ฝั่ง

- ฝั่งซ้ายเป็น `showcase panel` อธิบายคุณค่าของระบบ
- ฝั่งขวาเป็น `auth card` สำหรับเข้าสู่ระบบ

ช่องทางเข้าสู่ระบบที่รองรับ

- `เข้าสู่ระบบด้วย Google`
- `เข้าสู่ระบบด้วยอีเมล`

สารที่ระบบสื่อในหน้าล็อกอิน

- ระบบปลอดภัยและเปิดให้เฉพาะผู้ยืนยันตัวตนแล้ว
- ผู้ใช้จะเห็นภาพรวมตารางและภาระงานครูได้เร็ว
- ระบบพร้อมใช้งานจริง ไม่ใช่เพียง mockup

### 5.3 App Shell

หลังเข้าสู่ระบบ ผู้ใช้จะเข้าสู่ shell หลักของแอป ซึ่งประกอบด้วย

- `Sidebar` สำหรับ brand และสถานะระบบ
- `Topbar` สำหรับชื่อโรงเรียน ภาคเรียน หน้าปัจจุบัน ปุ่มรีเฟรช และโปรไฟล์ผู้ใช้
- `Navigation bar` สำหรับสลับเมนูหลัก
- `Workspace area` สำหรับเนื้อหาของแต่ละเมนู

ส่วนโปรไฟล์ผู้ใช้รองรับ

- avatar แบบย่อชื่อ
- ชื่อผู้ใช้
- อีเมล
- สถานะ provider เช่น Google หรือ Email
- การตั้งชื่อที่แสดงในระบบ
- ปุ่มออกจากระบบ

## 6. เมนูภาพรวม (Dashboard)

เมนู `ภาพรวม` ทำหน้าที่เป็น control room ของระบบ

### 6.1 Hero Section

เป็นพื้นที่นำสายตาของหน้า dashboard ใช้สื่อสถานะรวมของการจัดตาราง พร้อมปุ่ม action หลัก 2 ปุ่ม

- `จัดตารางอัตโนมัติ`
- `เปิดหน้าตารางสอน`

### 6.2 Dashboard Filter

รองรับการกรองตามระดับชั้น

- ทุกระดับชั้น
- ประถมศึกษา
- มัธยมศึกษาตอนต้น

### 6.3 KPI Metrics

การ์ดตัวเลขหลัก 5 ตัวประกอบด้วย

- จำนวนครู
- จำนวนห้อง/สถานที่
- จำนวนรายวิชา
- จำนวนคาบที่จัดแล้ว
- เปอร์เซ็นต์ความพร้อม

### 6.4 Insight Panels

ส่วนนี้ช่วยให้ผู้บริหารหรือฝ่ายวิชาการอ่านสถานะเชิงคุณภาพได้เร็ว

- วงแหวนความพร้อมของตาราง
- กราฟชั้นเรียนที่ควรติดตามต่อ
- รายการความครอบคลุมตามห้องเรียน
- รายการประเด็นที่ต้องตรวจสอบ
- สรุปภาระงานครูที่ควรจับตา

ข้อมูลที่ dashboard ใช้จริงมาจากผล `validation`, `teacherLoads`, `sectionStatuses`, `unresolvedGroups` และ `activity`

## 7. เมนูข้อมูลหลัก (Catalog)

เมนู `ข้อมูลหลัก` คือพื้นที่จัดการ master data ของระบบ โดยมีแนวคิดว่า “ข้อมูลพื้นฐานต้องถูกก่อน ตารางจึงจะถูกตาม”

องค์ประกอบของหน้านี้

- ตัวเลือกประเภทข้อมูล
- ตัวกรองเฉพาะประเภทข้อมูล
- ช่องค้นหา
- ปุ่ม `เพิ่มรายการ`
- ตารางข้อมูลพร้อมปุ่ม `แก้ไข` และ `ลบ`
- modal สำหรับกรอกหรือแก้ไขข้อมูล

เมนูย่อยใน catalog มี 6 ส่วน

### 7.1 ครู

บทบาทของเมนูนี้

- สร้างและแก้ไขข้อมูลครูผู้สอน
- กำหนดเพดานภาระสอนต่อสัปดาห์
- กำหนดบทบาทในระบบ
- ผูกรายวิชาที่ครูสามารถสอนได้

คอลัมน์ในตาราง

- รหัส
- ชื่อครู
- คาบสูงสุด
- บทบาท
- รายวิชา
- จัดการ

ตัวกรอง

- ผูกวิชาแล้ว
- ยังไม่ผูกวิชา
- มีบทบาทกำกับ
- ยังไม่กำหนดบทบาท

ฟิลด์ในฟอร์ม

- รหัสครู
- ชื่อ-สกุล
- คาบสอนสูงสุดต่อสัปดาห์
- บทบาทของผู้ใช้
- รายวิชาที่สอน

### 7.2 ห้อง/สถานที่

บทบาทของเมนูนี้

- จัดการห้องเรียนปกติ ห้องเฉพาะ และสถานที่ใช้งานจริง
- ใช้เป็นฐานข้อมูลสำหรับการกันห้องชนกันตอนจัดคาบ

คอลัมน์ในตาราง

- รหัส
- ห้อง/สถานที่
- ประเภท
- ความจุ
- จัดการ

ฟิลด์ในฟอร์ม

- รหัสห้อง/สถานที่
- ชื่อห้อง/สถานที่
- ประเภทห้อง
- ความจุ

### 7.3 รายวิชา

บทบาทของเมนูนี้

- กำหนดข้อมูลวิชาตามหลักสูตร
- รองรับทั้งรายวิชาทั่วไปและกิจกรรมพัฒนาผู้เรียน
- ใช้ข้อมูลนี้เป็นฐานในการคำนวณคาบและจัดหมวดหมู่

คอลัมน์ในตาราง

- รหัส
- รายวิชา
- หมวด/ประเภท
- หน่วยกิต
- คาบ/สัปดาห์
- จัดการ

ตัวกรอง

- ประเภทรายวิชา
- กิจกรรมพัฒนาผู้เรียน
- ประเภทกิจกรรม
- กลุ่มสาระ

ฟิลด์ในฟอร์ม

- หมวดข้อมูล
- รหัสวิชา
- ชื่อรายวิชา
- ประเภทรายวิชา
- หน่วยกิต
- คาบต่อสัปดาห์
- กลุ่มสาระการเรียนรู้
- ประเภทกิจกรรมพัฒนาผู้เรียน

### 7.4 ชั้นเรียน

บทบาทของเมนูนี้

- นิยามห้องเรียนเป้าหมายที่ต้องมีตาราง
- แยกระดับการศึกษา ปี และห้อง
- ระบุคาบเป้าหมายต่อสัปดาห์และครูประจำชั้น

คอลัมน์ในตาราง

- ระดับชั้น
- ชั้นเรียน
- ภาคเรียน
- คาบเป้าหมาย
- ครูประจำชั้น
- จัดการ

ตัวกรอง

- ทุกระดับชั้น
- ประถมศึกษา
- มัธยมศึกษาตอนต้น

ฟิลด์ในฟอร์ม

- ระดับชั้น
- ชั้นปี
- เลขห้อง/กลุ่มเรียน
- คาบเป้าหมายต่อสัปดาห์
- ปีการศึกษา
- ภาคเรียน
- ครูประจำชั้น

จุดเด่นเชิง UX

- มี preview ชื่อห้องเรียนให้ทันที เช่น `ป.4/1` หรือ `ม.2/A`
- เปลี่ยน grade จาก free text เป็น dropdown เพื่อลดความผิดพลาดตอนบันทึก

### 7.5 แผนรายวิชา

บทบาทของเมนูนี้

- กำหนดว่าวิชาใดเปิดสอนในห้องใด
- ระบุครูหลักและจำนวนคาบที่ต้องจัด
- ระบุห้องหรือสถานที่หลักของวิชานั้น

คอลัมน์ในตาราง

- ชั้นเรียน
- รายวิชา
- ครูผู้สอนหลัก
- คาบที่ต้องจัด
- ห้อง/สถานที่หลัก
- จัดการ

ตัวกรอง

- ทุกแผนรายวิชา
- ข้อมูลพร้อมจัดตาราง
- ยังไม่กำหนดครูหลัก
- ยังไม่กำหนดห้องหลัก

ฟิลด์ในฟอร์ม

- ชั้นเรียนที่เปิดสอน
- รายวิชา
- ครูผู้สอนหลัก
- คาบที่ต้องจัด
- ห้อง/สถานที่หลัก
- หมายเหตุ

### 7.6 กลุ่มการสอน

บทบาทของเมนูนี้

- แปลงแผนรายวิชาให้เป็นหน่วยการจัดจริง
- รองรับทั้ง `WHOLE_CLASS`, `SPLIT_GROUP`, `TEAM_TEACHING`, `LARGE_GROUP`
- กำหนดครูหลายคนต่อหนึ่งกลุ่ม พร้อมบทบาทและ load factor

คอลัมน์ในตาราง

- กลุ่ม
- รายวิชา
- ชั้นเรียน
- รูปแบบ
- ครู
- จัดการ

ตัวกรอง

- รูปแบบการสอน
- กำหนดครูแล้ว
- ยังไม่กำหนดครู

ฟิลด์ในฟอร์ม

- แผนรายวิชา

## 8. เมนูตารางสอน (Timetable Workspace)

นี่คือหน้าหลักของระบบในเชิงปฏิบัติการ และเป็นหัวใจของทั้งโปรเจกต์

### 8.1 โหมดมุมมอง

รองรับ 2 มุมมองหลัก

- `มุมมองห้องเรียน`
- `มุมมองครู`

ผู้ใช้สามารถสลับได้จาก segmented control และเลือก scope ที่ต้องการดูจาก dropdown

### 8.2 Toolbar หลัก

ประกอบด้วย action สำคัญในการทำงานจริง

- เลือกมุมมอง
- เลือกห้องหรือครู
- `ตรวจสอบ`
- `ส่งออก CSV`
- `ส่งออก PDF`
- `พิมพ์`

### 8.3 Export Console

ก่อนส่งออก ผู้ใช้สามารถกำหนดขอบเขตข้อมูลได้ 3 แบบ

- `current` ส่งออกเฉพาะรายการที่กำลังเปิดดู
- `selected` เลือกหลายห้องหรือหลายครู
- `all` ส่งออกทั้งหมดตามมุมมองปัจจุบัน

รองรับการค้นหาและเลือกชุดข้อมูลที่ต้องการส่งออกหรือพิมพ์

### 8.4 Board Layout

ตารางแสดงผลเป็น matrix แบบ `5 วัน x 6 คาบ`

- คอลัมน์คือวันเรียน
- แถวคือคาบเรียน
- แต่ละ cell รองรับหลาย entry ในกรณี split group

ข้อมูลที่แสดงบน card ของคาบ

- ชื่อวิชา
- ชื่อกลุ่มการสอน
- รายชื่อครูพร้อมบทบาท
- ห้องเรียนหรือสถานที่
- รูปแบบการสอน
- ปุ่มลบคาบออกจากตาราง

### 8.5 Group Pool

ด้านข้างของ board จะแสดง `ภาระสอนที่ยังเหลือ`

ในแต่ละ card ผู้ใช้จะเห็น

- ชื่อวิชา
- จำนวนคาบที่เหลือเทียบกับจำนวนคาบทั้งหมด
- ชื่อกลุ่ม
- ชั้นเรียน
- รูปแบบการสอน
- student group key
- ครูที่เกี่ยวข้อง

การทำงานของส่วนนี้

- คลิกที่กลุ่มเพื่อให้ระบบแนะนำคาบที่เหมาะสม
- ลาก card จาก group pool ไปวางบน board ได้โดยตรง

### 8.6 Inspector และ Collaboration Panel

ฝั่งขวาของ workspace รวมข้อมูลสนับสนุนการตัดสินใจ

- `คาบที่แนะนำ`
- `ผู้ใช้งานออนไลน์`
- `รายการล็อกการแก้ไข`
- `กิจกรรมล่าสุด`

ค่านี้ไม่ได้เป็นของตกแต่ง แต่เชื่อมกับ logic collaboration จริงจาก backend

### 8.7 Validation Panel

ส่วนล่างของหน้าจอแสดงผลตรวจตารางล่าสุด

- error
- warning
- คำอธิบายที่อ่านเข้าใจได้

ผู้ใช้สามารถกด `ตรวจสอบ` เพื่อขอให้ระบบประเมินตารางอีกครั้งหลังแก้ไข

### 8.8 พฤติกรรมการใช้งาน

รองรับ interaction หลัก 3 แบบ

- เลือก suggestion แล้วให้ระบบลงคาบ
- ลากกลุ่มที่ยังไม่ลงคาบไปวางใน slot ที่ต้องการ
- ลาก entry เดิมเพื่อย้ายคาบ

ทุกการเปลี่ยนแปลงในตารางจะไม่เขียนลงข้อมูลทันทีแบบไร้เงื่อนไข แต่จะผ่านกระบวนการ lock และ mutation ที่มี version กำกับ

## 9. เมนูตั้งค่า (Settings)

เมนูนี้เน้นการเตรียมข้อมูลสำหรับหัวเอกสารและการส่งออก

ฟิลด์สำคัญ

- ชื่อสถานศึกษา
- ชื่อย่อสถานศึกษา
- ปีการศึกษา
- ภาคเรียน
- โลโก้โรงเรียน
- ผู้ลงนาม 3 ชุด
- ตำแหน่ง ชื่อ-สกุล และภาพลายเซ็นของผู้ลงนามแต่ละคน

บทบาทของเมนูนี้คือเชื่อมระหว่างข้อมูลเชิงระบบกับเอกสารที่พิมพ์ใช้งานจริง ทำให้ PDF ที่ส่งออกไม่ใช่เพียงตารางเปล่า แต่มี identity ของสถานศึกษาและพร้อมใช้เป็นเอกสารทางงานวิชาการ

## 10. Workflow การทำงานของผู้ใช้

ลำดับการใช้งานที่เหมาะสมของระบบคือ

1. เข้าสู่ระบบด้วย Google หรือ Email
2. ตรวจสอบข้อมูลโรงเรียนและปีการศึกษาในเมนูตั้งค่า
3. เพิ่มข้อมูลครู ห้อง รายวิชา และชั้นเรียนในเมนูข้อมูลหลัก
4. สร้างแผนรายวิชาและกลุ่มการสอน
5. เปิดเมนูตารางสอนเพื่อเริ่มจัดคาบ
6. ใช้ suggestion, drag and drop หรือ auto-schedule เพื่อวางคาบ
7. ตรวจผล validation
8. ส่งออก CSV, PDF หรือพิมพ์เอกสาร

## 11. Logic การจัดตารางและกฎตรวจสอบ

ระบบมี validation engine จริงใน backend ไม่ได้พึ่งเพียงการเช็กระดับ UI

### 11.1 กฎ conflict หลัก

- `ROOM_DOUBLE_BOOKED` ห้องถูกใช้งานซ้อนกัน
- `TEACHER_DOUBLE_BOOKED` ครูสอนชนกันในช่วงเวลาเดียวกัน
- `SECTION_WHOLE_CLASS_COLLISION` คาบแบบทั้งห้องชนกับคาบอื่นใน slot เดียวกัน
- `SECTION_GROUP_OVERLAP` กลุ่มย่อยซ้ำกันใน slot เดียวกัน
- `TEACHER_OVERLOAD` ภาระสอนของครูเกินเพดาน
- `MISSING_SUBJECT_PERIODS` วิชายังถูกจัดไม่ครบตามจำนวนคาบ
- `MISSING_GROUP_PERIODS` กลุ่มการสอนยังถูกจัดไม่ครบ
- `SECTION_WEEKLY_TOTAL_INVALID` ห้องเรียนยังไม่ครบจำนวนคาบต่อสัปดาห์
- `SECTION_DAILY_TOTAL_INVALID` จำนวนคาบต่อวันของห้องไม่สมดุลตามที่กำหนด

### 11.2 Logic การแนะนำ slot

ระบบแนะนำ slot โดยใช้ score-based heuristic

- เริ่มจากคะแนนพื้นฐาน 100
- บวกคะแนนเมื่อช่วยเติมวันเรียนให้ครบ
- ลดคะแนนเมื่อครูมีคาบติดกันอยู่แล้ว
- ลดคะแนนเมื่อวิชานั้นเริ่มกระจุกในวันเดียว
- ลดคะแนนเล็กน้อยในช่วงปลายวัน
- ตัด slot ที่ครูชนหรือห้องชนออกตั้งแต่ต้น

ผลลัพธ์คือระบบจะคืน slot ที่เหมาะสมที่สุดประมาณ 8 รายการ เพื่อให้ผู้ใช้เลือกอย่างมีข้อมูล

### 11.3 Logic ของ Auto Schedule

Auto scheduler ใน implementation ปัจจุบันมีหลักทำงานดังนี้

- เรียงกลุ่มการสอนตามจำนวนคาบที่ต้องจัดและจำนวนครูในกลุ่ม
- ถ้ากลุ่มยังไม่มีครู ระบบจะ mark เป็น unresolved
- เรียก suggestion engine เพื่อหาคาบที่เหมาะสม
- พยายามหลีกเลี่ยงการลงกลุ่มเดิมซ้ำในวันเดียวมากเกินไป
- วางคาบทีละรายการจนกว่าจะครบหรือจนไม่พบ slot ที่ผ่านเงื่อนไข

นี่เป็นแนวทางกึ่งอัตโนมัติที่สมดุลระหว่างความง่ายในการอธิบายกับความสามารถในการใช้งานจริง

## 12. การทำงานร่วมกันหลายคน

หนึ่งในส่วนที่แข็งแรงของโปรเจกต์นี้คือระบบ collaboration ที่เชื่อมกับ workflow จริง

องค์ประกอบหลัก

- `presence` สำหรับบอกว่าใครออนไลน์อยู่
- `heartbeat` ทุก 60 วินาทีเพื่อคงสถานะผู้ใช้งาน
- `slot lock` สำหรับจองทรัพยากรที่กำลังแก้
- `expectedLockIds` ใน mutation เพื่อยืนยันว่าผู้แก้ถือ lock อยู่จริง
- `baseVersion` เพื่อกัน stale write
- `recent events` เพื่อแสดงกิจกรรมล่าสุดของผู้ใช้และระบบ

ทรัพยากรที่ระบบล็อกเวลาแก้คาบ

- ห้องเรียน
- กลุ่มการสอน
- ห้องหรือสถานที่
- ครูทุกคนที่อยู่ใน patch นั้น

แนวทางนี้เหมาะกับงานวิชาการที่มีหลายคนช่วยกันจัดตาราง เพราะลดความเสี่ยงจากการแก้ชนกันแบบเงียบ

## 13. การส่งออกและการพิมพ์

ระบบรองรับเอกสาร 3 รูปแบบการใช้งาน

- `CSV` สำหรับเปิดใน Excel หรือส่งต่อระบบอื่น
- `PDF` สำหรับแจกหรือจัดเก็บเป็นเอกสาร
- `Print` สำหรับสั่งพิมพ์จากเบราว์เซอร์

ความสามารถที่รองรับ

- ส่งออกรายครู
- ส่งออกรายห้องเรียน
- ส่งออกเฉพาะรายการที่เลือกหลายชุด
- ใส่ชื่อโรงเรียน โลโก้ ภาคเรียน ปีการศึกษา และรายชื่อผู้ลงนามลงใน PDF

## 14. สถาปัตยกรรมระบบจริงใน Repository

### 14.1 Frontend

Implementation ปัจจุบันใช้

- `index.html`
- `app.js`
- `render.js`
- `styles.css`
- `tailwind.css`

นั่นหมายความว่าเวอร์ชันที่อยู่ใน repo ตอนนี้เป็น `Vanilla JavaScript SPA` ไม่ใช่ React

### 14.2 Backend

Backend ใช้ Node.js แบบ lightweight

- server entry อยู่ที่ `server.js`
- ใช้ `node:http` สำหรับเสิร์ฟ API และ static files
- ใช้ runtime modules แยก concern เช่น scheduling, collaboration, export, storage

### 14.3 Authentication

ระบบยืนยันตัวตนด้วย Firebase Authentication

- Google sign-in
- Email/password sign-in
- bearer token แนบไปกับ API calls

### 14.4 Data Layer

ข้อมูล runtime ปัจจุบันเก็บใน `data/teachtable-db.json`

นอกจากนี้ยังมีโครงสำหรับ

- Prisma
- PostgreSQL adapter
- schema สำหรับขยายสู่ฐานข้อมูลจริงในอนาคต

### 14.5 Export Layer

- CSV export สร้างจาก report payload
- PDF export สร้างจาก runtime exporter
- รองรับ logo และ signatories จาก settings

## 17. สรุป

TeachTable เป็นโปรเจกต์ที่มีทิศทางชัดเจนทั้งด้าน product, UX และ architecture โดยวางตำแหน่งตัวเองเป็นระบบจัดตารางสอนที่ “พร้อมใช้จริง” มากกว่าหน้า mockup ทั่วไป โครงสร้างเมนู, การแบ่ง screen, การจัดลำดับข้อมูล, logic การตรวจชน, การทำงานร่วมกันหลายคน และการส่งออกเอกสาร ล้วนเชื่อมถึงกันอย่างเป็นระบบ
